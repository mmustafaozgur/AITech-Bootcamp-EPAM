# Research: User Authentication System

**Phase**: 0 — Research & Decisions  
**Feature**: 001-user-auth  
**Date**: 2026-05-08

---

## 1. TypeScript + Express.js Project Structure

**Decision**: CommonJS modules (not ESM), `"strict": true`, `ts-node-dev` for local dev.

**Rationale**: Express.js and most of its ecosystem (e.g., pg, bcrypt, jsonwebtoken) publish
CJS packages. ESM interop in Node.js/Express as of 2026 still requires awkward workarounds
(no `require`, extensions on imports, limited `__dirname` support). CJS + TypeScript strict mode
gives full type safety with zero interop friction. `ts-node-dev` provides hot-reload with no
build step during development; `tsc` produces the production output.

**Key tsconfig settings**:
```json
{
  "compilerOptions": {
    "strict": true,
    "target": "ES2020",
    "module": "CommonJS",
    "moduleResolution": "node",
    "outDir": "dist",
    "rootDir": "src",
    "resolveJsonModule": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "sourceMap": true
  }
}
```

**Alternatives considered**:
- ESM: Rejected — Express + pg + bcrypt ecosystem friction outweighs benefits at this project scale.
- NestJS: Rejected — Adds significant complexity (decorators, DI framework) beyond the feature scope.

## 2. Password Hashing — bcrypt

**Decision**: Use `bcrypt` (`bcrypt` npm package, native bindings) with cost factor 12.

**Rationale**: The user specification explicitly requires bcrypt. Cost factor 12 meets OWASP's
recommendation of ≥ 100ms hash time on current hardware and provides forward-compatibility
by being ratchetable. The native `bcrypt` package is faster than the pure-JS `bcryptjs`.

**Security note**: Argon2id (via `argon2` package) is the current OWASP first-choice (memory-hard,
resistant to GPU attacks). If this system is revisited for a security hardening sprint, Argon2id
should be evaluated as a drop-in replacement. bcrypt is still acceptable per OWASP 2023 guidance
with cost ≥ 10.

**Alternatives considered**:
- `bcryptjs`: Pure-JS (~30% slower); acceptable only if native bindings are unavailable.
- `argon2`: OWASP first-choice, but not specified by the user for this iteration.
- `PBKDF2` (built-in): No third-party dep but requires manual parameter management; lower security margin.

## 3. JWT Library — jose

**Decision**: Use `jose` (by Auth0) for signing and verifying JWTs.

**Rationale**: `jose` is TypeScript-first (no `@types/` needed), promise-based, and supports
HS256 (HMAC-SHA256) and RS256. It compiles cleanly under `strict: true` with no type assertions
needed. It is actively maintained and receives security patches promptly.

**Algorithm**: HS256 (shared secret) for this single-service deployment. RS256 would be chosen
if a separate resource server needed to verify tokens without access to the signing secret.

**Alternatives considered**:
- `jsonwebtoken`: Callback-based, requires `@types/jsonwebtoken`, outdated API; still widely used.
- `fast-jwt`: ~2× faster, but callback-based; speed is not a bottleneck here.
- `passport-jwt`: Passport framework overhead; rejected (see §8).

## 4. PostgreSQL Client — `pg` (node-postgres) + Drizzle ORM

**Decision**: Use `drizzle-orm` with the `pg` driver.

**Rationale**: Drizzle provides TypeScript-first schema definitions (the TypeScript file is the
single source of truth), generates readable SQL (no black-box query engine), and is lightweight
(~15 KB). It integrates directly with `pg` connection pools. Migrations are generated by
`drizzle-kit`. This gives strict type safety without the 100 MB+ overhead of Prisma or the
decorator coupling of TypeORM.

**Alternatives considered**:
- `pg` (raw SQL only): Full control but no compile-time query type safety; rejected in favour of Drizzle.
- `Prisma`: Excellent DX but 100 MB+ install, binary engine, and schema split from TypeScript; overkill for a single auth service.
- `TypeORM`: Decorator-heavy, slower, less active community than Drizzle.
- `Sequelize`: Legacy; poorer TypeScript support.

## 5. Rate-Limiting — In-Process Map

**Decision**: Use an in-process `Map<string, RateLimitEntry>` with TTL-based cleanup.

**Rationale**: The spec targets a single-region, single-instance deployment. An in-process map
adds zero network latency, has no external dependency, and is sufficient for the 500-user
concurrency target. A sliding-window counter keyed by email address implements the 5-attempts-per-
15-minutes rule from FR-006.

**Migration path**: If the service is horizontally scaled, swap the in-process map for Redis
(`ioredis`) — the service interface stays the same.

**Alternatives considered**:
- `express-rate-limit` (in-memory store): Library wrapper around the same in-process pattern;
  adding a library for ~40 lines of code is unnecessary.
- `ioredis`: Correct for multi-instance; over-engineered for this scope.

## 6. Testing — Jest with ts-jest

**Decision**: Use `jest` with `ts-jest` transformer.

**Rationale**: The user specification requires Jest. `ts-jest` compiles TypeScript inline during
test runs without a separate build step, supports `strict: true`, and requires minimal
configuration. Test coverage is collected by Jest's built-in Istanbul integration.

**Jest config (`jest.config.ts`)**:
```ts
import type { Config } from 'jest';

const config: Config = {
  preset: 'ts-jest',
  testEnvironment: 'node',
  testMatch: ['**/*.test.ts'],
  collectCoverageFrom: ['src/**/*.ts', '!src/**/*.d.ts', '!src/index.ts'],
  coverageThreshold: { global: { lines: 80, branches: 80 } },
  moduleNameMapper: { '^@/(.*)$': '<rootDir>/src/$1' },
};

export default config;
```

**Alternatives considered**:
- `@swc/jest`: 10–100× faster compilation, but less stable with some `ts-jest`-specific mocking
  patterns; consider if test suite grows large and runtime becomes a bottleneck.
- `babel-jest`: Requires separate Babel config; no advantage here.

## 7. Email Sending — Nodemailer

**Decision**: Use `nodemailer` with an SMTP transport.

**Rationale**: Nodemailer is the de-facto Node.js SMTP library. For local development, it works
with [Ethereal Email](https://ethereal.email) (free catch-all SMTP) or a local MailHog instance —
no real emails sent. In production, the SMTP credentials point to any transactional provider
(SendGrid, AWS SES, Resend). This approach requires no SDK changes when switching providers.

**Alternatives considered**:
- `@sendgrid/mail`: Provider lock-in; heavier SDK.
- `resend`: Excellent DX, but requires an API key even for development.
- `AWS SES SDK`: Complex setup; overkill for this scope.

## 8. Auth Middleware — Custom Express Middleware

**Decision**: Implement a custom `authenticate` middleware function (~30 lines).

**Rationale**: The system uses a single auth strategy (JWT Bearer token). Passport.js is designed
for multi-strategy scenarios (OAuth, SAML, local, JWT, etc.) and adds ~100 KB of abstraction,
awkward TypeScript typings, and debugging complexity. A custom middleware is transparent, fully
typed, and trivially testable.

**Alternatives considered**:
- `passport` + `passport-jwt`: Rejected — unnecessary abstraction for a single JWT strategy.
- `express-jwt`: Wraps `jsonwebtoken`; rejected (using `jose`).

## 9. REST Endpoint Conventions

**Decision**: All auth routes under `/auth` prefix; all state-changing operations use `POST`.

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/auth/register` | Create account |
| `POST` | `/auth/login` | Issue access + refresh token |
| `POST` | `/auth/refresh` | Exchange refresh token for new access token |
| `POST` | `/auth/logout` | Revoke refresh token |
| `POST` | `/auth/verify-email` | Confirm email via token in link |
| `POST` | `/auth/forgot-password` | Request password-reset email |
| `POST` | `/auth/reset-password` | Set new password via reset token |
| `DELETE` | `/auth/account` | Delete account (right to erasure, FR-013) |
| `GET` | `/auth/me` | Return current user profile (protected) |

**Rationale**: `POST` for all write operations follows REST semantics. `GET /auth/me` is the
only safe idempotent call. `DELETE /auth/account` uses the correct HTTP verb for resource deletion.

**Alternatives considered**:
- `GET /logout`: Rejected — logout has a side effect (token revocation); must be `POST`.
- `PUT /auth/reset-password`: `POST` is conventional for token-exchange operations.

## 10. Environment Variable Management — dotenv + zod

**Decision**: Use `dotenv` for loading `.env` files, with a `src/config/env.ts` module that
validates all variables using `zod` at startup.

**Rationale**: `dotenv` is the universal Node.js standard with zero magic. Wrapping it in a `zod`
schema provides the startup-fail-fast behaviour and type narrowing that `@t3-oss/env-core` offers,
without adding another opinionated dependency. The pattern is ~20 lines and fully transparent.

```typescript
// src/config/env.ts
import { z } from 'zod';
import 'dotenv/config';

const schema = z.object({
  NODE_ENV: z.enum(['development', 'test', 'production']).default('development'),
  PORT: z.coerce.number().default(3000),
  DATABASE_URL: z.string().url(),
  JWT_SECRET: z.string().min(32),
  BCRYPT_ROUNDS: z.coerce.number().default(12),
  SMTP_HOST: z.string(),
  SMTP_PORT: z.coerce.number().default(587),
  SMTP_USER: z.string(),
  SMTP_PASS: z.string(),
  BASE_URL: z.string().url(),
});

export const env = schema.parse(process.env);
```

**Alternatives considered**:
- `@t3-oss/env-core`: Excellent but pulls in Zod as a peer dep and opinionated structure; unnecessary for a single service.
- `dotenv-safe`: Validates against `.env.example` (manual sync); less type-safe.
- `plain process.env`: No validation; silent `undefined` bugs at runtime.
